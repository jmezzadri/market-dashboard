# Integration Roadmap: Market Stress Dashboard + Trading Signal Scanner
**Prepared for Joe Mezzadri | April 14, 2026**

---

## The Big Picture

You currently have two independent, working tools:

**Market Stress Dashboard** — A macro top-down system that tracks 25 financial indicators, computes a composite stress score, maps it to a regime (LOW / NORMAL / ELEVATED / EXTREME), and emails a daily AI narrative. It tells you *what kind of market environment you're in*.

**Trading Signal Scanner** — A bottom-up signal engine that ingests options flow, dark pool, congressional and insider trades, scores tickers 0–100, screens covered call opportunities, and emails alerts when high-conviction signals fire. It tells you *which stocks to act on*.

The integration goal is simple: **one unified daily email** that answers all three questions in sequence:
1. What is the macro environment saying right now? (Dashboard)
2. Given that backdrop, what specific opportunities are firing? (Scanner)
3. What do I need to do with my existing positions today? (Portfolio)

This sequencing matters. A 75/100 BUY signal in an EXTREME stress regime should be treated very differently than the same signal in a LOW stress regime. Right now, you'd have to read two separate emails and mentally connect them. The integrated tool does that for you.

## Integration Architecture

The integration works by adding a **coordination layer** — a single orchestration script (`morning_brief.py`) that:

1. Reads the current stress regime from `src/App.jsx` (already parsed by the dashboard)
2. Runs the trading scanner's signal pipeline
3. Passes the regime context to Claude when generating the narrative
4. Builds one combined HTML email in a defined section order
5. Sends at the scheduled time (replacing the two separate emails)

```
7:00 AM  →  fetch_indicators.py runs (dashboard data fetch + App.jsx patch + git push)
7:15 AM  →  morning_brief.py runs:
              ├── reads composite stress score + regime from App.jsx
              ├── runs trading scanner pipeline (UW signals + scoring + CC screening)
              ├── pulls Schwab positions + cash + open calls
              ├── calls Claude with combined context → unified narrative
              └── sends one HTML email
```

The scanner's intraday and post-market emails continue running independently — they're alert-driven, not summary-driven. The integration only replaces the morning email cadence.

## Unified Email Structure

The combined daily email would be structured as follows:

### Subject line
`Market Brief — [Date] | Regime: ELEVATED ↑ | 2 Buy Signals`

### Section 1: Market Regime Header (3 lines)
Composite stress score (e.g., 58/100), regime label, trend direction vs. 1-month prior. Color-coded. This is the first thing you see — the macro temperature before any trade ideas.

### Section 2: AI Macro Narrative (4 paragraphs)
The same structured analysis the dashboard already generates: Regime Summary → What the Data Is Saying → Key Risks to Watch → Portfolio Implications. Generated by Claude using live indicator data. The key upgrade: the prompt explicitly instructs Claude to frame portfolio implications in terms of whether the current regime supports adding new positions and at what sizing.

### Section 3: Trade Signals (conditional — only shown when signals exist)
Buy-tier signals (score ≥60) with regime-adjusted context. For example: "CCJ scores 72/100 — insider cluster + congressional buy. Note: regime is ELEVATED; consider half-sized entry and tighter stop." Watch-tier signals (35–59) listed below with a brief rationale.

### Section 4: Covered Call Opportunities
For any buy-tier signal where a valid covered call exists: strike, expiry, premium, annualized yield. Regime context surfaced here too — in EXTREME stress, this section would note that IV expansion may favor waiting rather than selling premium immediately.

### Section 5: Portfolio Alerts
Carry-forward of the scanner's existing sell/roll logic: covered calls approaching 80% max profit, positions triggering stop-loss, insider reversals, score collapses. Your current CCJ and OXY positions would always appear here if any action flags fire.

### Section 6: Weekly Addendum (Sundays only)
Full week recap: what changed in the stress score week-over-week, how scanner signals evolved, what positions were recommended vs. taken, next week's key catalysts and scheduled data releases.

## Regime × Signal Interaction Logic

This is the core value-add of integration. Here's the proposed decision table:

| Regime | Score ≥60 (Buy) | Score 35–59 (Watch) | Recommendation Posture |
|--------|-----------------|---------------------|----------------------|
| LOW | Full size entry | Elevated watchlist | Normal — execute as signaled |
| NORMAL | Full size entry | Standard watchlist | Normal — execute as signaled |
| ELEVATED | Half size, tighter stop | Watch only, no buy | Caution — regime constrains sizing |
| EXTREME | No new longs | Defer all | Hold — preserve capital, wait for regime shift |

This table gets embedded in Claude's prompt so the AI narrative automatically reflects the interaction — you don't need to mentally cross-reference two separate emails.

**Deliverable:** Single HTML email with all six sections above. Both separate morning emails retired.

## What Stays Separate

The intraday and post-market scanner alerts (10 AM, 12 PM, 2 PM, 4:30 PM) should remain independent. These are time-sensitive — unusual options flow at noon doesn't need to wait for a macro check. The regime context is relevant for sizing decisions, not for the alert itself. Those emails continue as-is.

The dashboard's `fetch_indicators.py` and Vercel deployment also stay independent — they have no dependency on the scanner and should run even on days when no trade signals fire.

---

## Dependencies and Risks

**Path coupling:** The orchestration script needs to know where both projects live on Joe's Mac. Proposed: use environment variables in `~/.config/market-dashboard.env` (which already exists) to set `TRADING_SCANNER_PATH=~/Developer/trading-scanner` or wherever the scanner lives.

**Timing:** The 7:15 AM brief assumes the 7:00 AM fetch completes in under 15 minutes. Historical logs show it runs in ~2 minutes. Safe buffer.

**API failures:** If the scanner's Unusual Whales call fails, the morning brief should still send the macro narrative. Graceful degradation is required — Phase 4 build should treat the scanner's output as optional sections.

**Schwab authentication:** The Schwab API uses OAuth with token refresh. If the token expires overnight, the portfolio section degrades gracefully to "Schwab unavailable — positions not refreshed."
